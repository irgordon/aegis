# AEGIS v0.4.1 Desktop First-Run Review

## Summary

Recommendation: PASS WITH FIXES.

The desktop application launches and renders a complete first screen. A first-time developer can see that AEGIS is showing governed execution evidence and that the backend remains authoritative.

The largest issue is release identity drift. The UI still presents `PRE-ALPHA`, `LOCAL-ONLY`, and `v0.4.0` labels even though the public release being evaluated is the `v0.4.1` Developer Preview. That mismatch weakens trust during the first ten seconds.

Visual evidence reviewed:

- Uploaded screenshot: `Screenshot 2026-07-04 at 15.07.58.png`
- Published release: `v0.4.1`
- Artifact: `aegis-v0.4.1-macos-arm64.tar.gz`

The screenshot was reviewed as user-provided evidence. It was not edited or committed.

## Launch

The desktop process launched from the published arm64 artifact.

No missing resources or broken rendering were visible in the screenshot. The UI rendered the sidebar, title area, status cards, timeline, live error card, recovery cards, and authority-boundary notice.

The published artifact README remains stale. It still describes the archive as a draft inspection artifact and uses pre-alpha/local-only wording. That is already tracked from the developer download review, but it also affects the desktop launch path because it is the first file a new user sees before opening the app.

## First Impression

Within the first ten seconds, the user can tell:

- the application is AEGIS
- it is about governed execution evidence
- live backend `health.check` evidence is present
- recovery evidence is sample-only

The first impression is weakened by conflicting release labels:

- `PRE-ALPHA`
- `LOCAL-ONLY`
- `v0.4.0`
- `v0.4.0 pre-alpha local-only`

Those labels conflict with the public `v0.4.1` Developer Preview release page and README.

## Visual Hierarchy

The left navigation and large `AEGIS` heading create a clear identity anchor.

The evidence source banner is prominent and useful. The current execution status cards are easy to scan. The timeline is visually grouped and communicates a staged execution flow.

The right column draws attention because of strong colored borders. This helps recovery and error sections stand apart, but the red `Live Error Evidence` panel looks like an active error even when no live error is present.

## Information Density

The UI is still dense for a first-time developer.

The status cards and timeline communicate useful evidence, but the screen asks the user to process many governance terms at once. The right column adds more detail before the user knows which panels matter.

Status is immediately visible. Meaning takes longer.

The best part of the density is that live and sample evidence are labeled. The weakest part is that there is no single next step after the launch.

## Terminology

Several visible terms are precise but not beginner-friendly:

- Policy Bundle
- Policy Decision
- Credential Boundary
- Credential Injection
- Wrapper Execution
- Audit Log
- State Log
- Recovery Inspection
- Recovery Plan
- Evidence

A developer familiar with execution governance can infer most of these. A first-time evaluator may need hover text or a simpler first-run mode.

The labels `Live backend health.check evidence` and `Sample evidence remains labeled` are clear and useful.

## Status Communication

The UI clearly communicates:

- live backend `health.check` evidence is available
- sample recovery evidence remains labeled
- credential injection is not required
- execution completed
- the UI is not an authority boundary

The status communication problem is release identity. The UI says `v0.4.0`, `PRE-ALPHA`, and `LOCAL-ONLY` while the product is being evaluated as public `v0.4.1` Developer Preview.

## Call to Action

The first screen does not answer "what should I do next?"

The UI shows what happened, but it does not guide the developer toward the next evaluation step. A first-time user may not know whether to click navigation items, inspect the gateway output, run the smoke test, or read the artifact README.

The UI should provide one small next step without turning the screen into a tutorial.

## Cognitive Load

The UI reduces execution risk by making evidence visible, but it still asks the user to learn many AEGIS terms at once.

The screen is understandable after careful reading. It is not yet effortless to scan.

The most useful cognitive-load reducers are already present:

- clear live/sample labels
- visible stage outcomes
- authority-boundary notice
- status cards before details

The main load comes from dense terminology, duplicated status evidence, and sample recovery sections competing with the live execution path.

## Accessibility

The primary navy text on the cream background is readable. Layout spacing is adequate, and the main status cards are large enough to scan.

Accessibility risks:

- yellow sample labels on a pale background may have weak contrast
- the sidebar footer text is small
- some card labels wrap tightly, especially `Credential Injection`
- the UI uses color heavily to distinguish live, sample, warning, and error states
- the red error panel can be misread as an active failure

No obvious clipping or truncation was visible in the screenshot.

## Usability Findings

| ID | Severity | Observation | Recommendation |
| --- | --- | --- | --- |
| AEGIS-DFR-001 | P1 | The desktop UI shows `PRE-ALPHA`, `LOCAL-ONLY`, and `v0.4.0` during a `v0.4.1` Developer Preview evaluation. | Update desktop release identity labels to match the public Developer Preview release. |
| AEGIS-DFR-002 | P2 | The first screen shows evidence but does not give a clear next step. | Add a small first-run action or hint that points to the bundled gateway smoke test or artifact README. |
| AEGIS-DFR-003 | P2 | The red `Live Error Evidence` panel appears alarming even when no error exists. | Use a neutral empty state when no live error report is present. |
| AEGIS-DFR-004 | P2 | Governance terms are accurate but heavy for a new evaluator. | Add concise hover explanations or simplify labels in first-run mode. |
| AEGIS-DFR-005 | P2 | Navigation labels imply deeper functional sections, but the first screen does not explain what is live versus future. | Label inactive or future sections clearly, or keep first-run navigation focused on available evidence. |
| AEGIS-DFR-006 | P3 | Yellow and orange sample labels may be hard to read against the cream background. | Check contrast and adjust sample-state colors if needed. |
| AEGIS-DFR-007 | P3 | The published artifact README is stale and still says draft inspection artifact, pre-alpha, and local-only. | Refresh published artifact contents in the next Developer Preview artifact build. |

## Prioritized Improvements

1. Update desktop release identity to `v0.4.1 Developer Preview`.
2. Replace the red no-error card with a neutral no-error state.
3. Add a first-run next step that points to the bundled gateway smoke test.
4. Add short explanations for heavy governance terms.
5. Review contrast for sample-only labels and sidebar footer text.
6. Refresh artifact README content in the next public artifact.

## Follow-up Status

The highest-friction desktop findings were fixed in source after this review.

- `AEGIS-DFR-001`: fixed. The UI now shows `v0.4.1`, `Developer Preview`, `Public prerelease`, `Unsigned`, and `Not notarized`.
- `AEGIS-DFR-002`: fixed. The first screen now includes a small next-step cue that points to the archive README and gateway smoke test.
- `AEGIS-DFR-003`: fixed. The no-error state is neutral unless live backend error evidence exists.
- `AEGIS-DFR-004`: narrowed. First-screen labels now use shorter terms such as `Policy`, `Credentials`, `Wrapper`, `Audit`, `State`, and `Recovery`.

Artifact-level verification remains tied to the next generated Developer Preview artifact.

## Overall Evaluation

Would I continue evaluating this product?

Yes. The UI launched and showed enough real governed evidence to continue.

Would I understand its purpose?

Mostly. The purpose is visible, but stale release labels and heavy terms create avoidable doubt.

Would I recommend another developer try it?

Yes, with the caveat that the first-run UI needs a release-label update and clearer next-step guidance.

## Recommendation

PASS WITH FIXES.

The desktop is launchable and communicates that AEGIS renders governed backend evidence. It is good enough to continue Developer Preview evaluation.

Before treating the desktop first-run experience as polished, fix the stale release labels and reduce the chance that a user misreads the no-error panel as an active failure.
