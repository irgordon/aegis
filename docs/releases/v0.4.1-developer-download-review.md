# AEGIS v0.4.1 Developer Download Review

## Summary

Recommendation: PASS WITH FIXES.

The public `v0.4.1` Developer Preview can be found, downloaded, verified, extracted, and launched from the published GitHub Release.

The release artifacts are intact. When both macOS archives and `SHA256SUMS` are present, checksum verification passes.

The first-run experience still has friction. The most important issues are checksum instructions for single-archive downloads, stale README text inside the published archive, and lack of a complete gateway smoke-test path inside the artifact.

Desktop visual inspection was environment-blocked because the review was run remotely from an iPhone while the monitor was off. The desktop binary launched and stayed running, but the captured screen was black. This review does not treat that as a product failure.

## Download Experience

The release is discoverable from the README link to the `v0.4.1` GitHub Release.

The release exposes three expected assets:

- `aegis-v0.4.1-macos-arm64.tar.gz`
- `aegis-v0.4.1-macos-x64.tar.gz`
- `SHA256SUMS`

A new macOS user can infer the likely archive from the architecture name, but the release page does not explain Apple Silicon versus Intel selection.

## Checksum Experience

The arm64 archive and `SHA256SUMS` were downloaded from the public release.

Running the documented full-manifest command with only the arm64 archive present produced:

```text
aegis-v0.4.1-macos-arm64.tar.gz: OK
aegis-v0.4.1-macos-x64.tar.gz: FAILED open or read
```

This is not an artifact integrity failure. It is a documentation friction issue. The manifest covers both macOS archives, so a user who downloads only one archive needs a selected-archive verification command.

When all three assets were downloaded, full verification passed:

```text
aegis-v0.4.1-macos-arm64.tar.gz: OK
aegis-v0.4.1-macos-x64.tar.gz: OK
```

## Extraction Experience

The arm64 archive extracted cleanly.

The extracted structure was clear:

```text
aegis-v0.4.1-macos-arm64/
  README.md
  ARTIFACT-CONTENTS.md
  bin/aegis-gateway
  desktop/aegis-desktop
  policy-bundles/local-dev/
```

The policy bundle includes the expected runtime files:

- `manifest.yaml`
- `gateway_policy.yaml`
- `risk_matrix.yaml`
- `checksums/SHA256SUMS`
- `signatures/public.pem`
- `signatures/SHA256SUMS.sig`

The archive README and content manifest in the published artifact are stale. They still describe the artifact as a draft inspection artifact and use older pre-alpha/local-only wording.

## Desktop Launch

The desktop binary was launched directly from the extracted archive:

```bash
./desktop/aegis-desktop
```

The process started and stayed running with no terminal error.

Visual inspection was blocked by the test environment. The review was run remotely from an iPhone while the monitor was off, and the captured screenshot was black.

Result: process launch PASS; visual first-screen review BLOCKED by environment.

## Gateway Launch

The gateway binary was launched directly from the extracted archive.

Running `./bin/aegis-gateway` returned a structured error with usage guidance.

Running `./bin/aegis-gateway --help` also returned the same structured error. A new developer may expect `--help` to print normal help text and exit successfully.

The archive does not include request fixture files. That means a new developer cannot run a complete successful gateway smoke test using only the downloaded artifact.

## Documentation Review

Public README and wiki overview explain what AEGIS is and link to the release.

Before this review, the public docs did not give a concise first-run path for:

- choosing arm64 versus x64
- verifying only one selected archive
- extracting the archive
- launching the desktop binary
- launching the gateway binary

This review added a focused first evaluation path to the wiki overview and linked it from the README.

## Usability Findings

| ID | Severity | Location | Observed friction | Recommended improvement |
| --- | --- | --- | --- | --- |
| AEGIS-DL-001 | P1 | Release checksum instructions | The full `shasum -a 256 -c SHA256SUMS` command fails when a user downloads only one platform archive. | Document selected-archive verification or tell users to download all assets before running the full manifest command. |
| AEGIS-DL-002 | P1 | Published archive README | The published artifact README says draft inspection artifact, pre-alpha, and local-only. | Refresh artifact README content in the next artifact build or patch release. |
| AEGIS-DL-003 | P1 | Gateway artifact | The archive has no request fixture, so the gateway binary cannot demonstrate a successful governed request from the artifact alone. | Include a safe health-check request fixture or documented minimal request in a future artifact. |
| AEGIS-DL-004 | P2 | Gateway CLI | `--help` returns an invalid-arguments error instead of normal help output. | Add conventional `--help` handling or document that running without arguments prints usage as a structured error. |
| AEGIS-DL-005 | P2 | Release page | Asset architecture selection is not explained on the release page. | Add plain language guidance for Apple Silicon versus Intel Macs. |
| AEGIS-DL-006 | P2 | Desktop launch review | Visual first-screen validation could not be completed because the review environment had the monitor off. | Re-run visual validation from a local display before treating first-screen clarity as fully verified. |

## Prioritized Improvements

1. Keep the selected-archive checksum command in public docs.
2. Refresh the artifact README in the next artifact build.
3. Add a safe request fixture or explicit artifact-only gateway smoke test.
4. Add conventional gateway help output.
5. Re-run desktop visual validation from an active display.

## Recommendation

PASS WITH FIXES.

The published Developer Preview is downloadable and the artifacts verify. The archive starts the desktop and gateway binaries from extracted contents. First-run usability should improve before expanding platform support.
